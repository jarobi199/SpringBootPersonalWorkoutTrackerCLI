01 / Overview
Project Overview

Application Summary

The Personal Workout Tracker is a menu-driven command-line application built with Java, Spring Boot, and MongoDB. You maintain a personal exercise library — a catalogue of exercises you define with name, muscle group, and equipment. From there you log workout sessions, recording sets, reps, and weight per exercise. Over time the app tracks your progression on each exercise and generates performance reports showing volume trends, strength gains, and weekly and monthly summaries.

The navigation follows the same two-level numbered menu pattern established in the Finance and Warehouse projects — a main menu dispatches to submenus, each focused on one domain area. The architecture uses the three-collection pattern: users, exercises, and sessions — mirroring the accounts, products, and movements structure you already know well.

Technology Stack

Spring Boot Entry point via CommandLineRunner. No web layer. Spring DI wires all services. SessionContext holds the current logged-in user.
Spring Data MongoDB Three repositories: UserRepository, ExerciseRepository, SessionRepository. Polymorphic _class discriminator for exercise subtypes. Sessions reference exercises by exerciseId.
Java OOP Abstract Exercise with three concrete subtypes each overriding calculateVolume(). Factory pattern for exercise creation. Chain of responsibility for alerts. Menu classes are Spring @Component beans wired via DI.
Menu System Two-level numbered menus. Scanner-based InputHandler with debug mode. do-while loop navigation. 0 always means Back or Exit.
BCrypt Passwords hashed via Spring Security Crypto. Plain-text passwords are never stored or logged.

Domain Model

Exercise Hierarchy

All exercise types extend the abstract Exercise class. The key differentiator is calculateVolume() — each subtype computes volume differently, giving a meaningful intensity metric for reports and progression tracking.

Core Classes

User id, name, username, hashedPassword, role (enum), bodyWeightKg, weeklySessionGoal. bodyWeightKg is needed by BodyweightExercise.calculateVolume() and stored on the user rather than on each session entry. Top-level MongoDB document.
Exercise (abstract) id, userId, name, muscleGroup (enum), equipment (enum), exerciseType (enum), notes. Abstract methods: calculateVolume(), getExerciseType(). Own MongoDB collection — each user maintains their own exercise library. Subtypes stored via _class discriminator.
StrengthExercise Extends Exercise. Adds no extra definition fields — sets, reps, and weight are recorded per session entry, not on the exercise definition itself.
CardioExercise Extends Exercise. Adds no extra definition fields — duration and distance are recorded per session entry.
BodyweightExercise Extends Exercise. Adds no extra definition fields — sets, reps, and optional added weight are recorded per session entry.
WorkoutSession id, userId, sessionDate, durationMinutes, notes, List<SessionEntry>. Own MongoDB collection. Immutable once completed. SessionEntry is embedded — not a top-level document.
SessionEntry Embedded in WorkoutSession. exerciseId, exerciseName (denormalized for display), exerciseType, sets, reps, weightKg, durationMinutes, distanceKm, addedWeightKg, notes. Fields are nullable — only the relevant ones are populated per exercise type.
AlertContext Carries data for alert evaluation: sessionEntry, workoutSession. Fields nullable — strategies check supports() to determine applicability. Transient — never persisted.
AlertManager Spring @Component. Auto-populated List<AlertStrategy>. Single evaluate(AlertContext) traverses all strategies. Called after each session entry is logged.

Alert Strategies

OvertrainingStrategy supports() — true when context.getWorkoutSession() != null. evaluate() — fires when the same primary muscle group has been trained on consecutive days. Warns the user that the muscle group may need rest. Called from SessionService when a session is completed.
GoalProgressStrategy supports() — true when context.getWorkoutSession() != null. evaluate() — fires when weekly session count drops below the user's configured target (default 3 sessions per week). Called from ReportService during report generation.

Key Design Notes

Exercise vs Entry The Exercise document in the exercise library defines what the exercise is — name, muscle group, equipment, type. The SessionEntry embedded in a session records what you actually did — sets, reps, weight, distance. An exercise can appear in hundreds of sessions; the entry captures the performance for that specific session only.
Denormalized Name SessionEntry stores exerciseName alongside exerciseId. This avoids a lookup when displaying session history — the name is already there. If an exercise is renamed the old session entries keep the original name, which is actually desirable for historical accuracy.
Volume Calculation calculateVolume() is abstract on Exercise but is actually called on the SessionEntry via the resolved exercise subtype. SessionService loads the exercise by id, casts to the correct subtype, and calls calculateVolume(entry) passing the entry's recorded values. Volume is never stored — always derived.
ExerciseFactory ExerciseFactory.create(ExerciseType, ...) prompts common fields (name, muscleGroup, equipment, notes) then type-specific ones. Since the exercise definition has no type-specific fields, the factory difference is only in which Exercise subtype is instantiated. The type determines how session entries are recorded in the session menus.
BodyweightExercise Volume BodyweightExercise.calculateVolume() needs the user's body weight. This is fetched from SessionContext.getCurrentUser().getBodyWeightKg() at calculation time. If the user has not set their body weight, volume defaults to sets × reps only (added weight still applied if present).

05 / Features
Feature Specifications

Exercises

List all exercisesDisplay all exercises in the library: name, type badge (STR / CRD / BWt), muscle group, equipment. Filterable by muscle group or equipment at the prompt.
Add exerciseSelect exercise type first. ExerciseFactory instantiates correct subtype. Prompt name, muscle group, equipment, and optional notes. Persist via ExerciseRepository.
View exercise detailFull definition fields plus last 5 session entries for this exercise and a mini ASCII sparkline (▁▂▃▄▅▆▇█) of the key metric over the last 8 sessions via SparklineUtil.renderLastN(values, 8).
Delete exerciseConfirm before deletion. Reject if any session entries reference this exercise — historical integrity must be preserved.

Sessions

Start new sessionCreates a new WorkoutSession with today's date. Prompts to add exercises one by one. For each exercise, prompts type-specific fields (sets/reps/weight for Strength, duration/distance for Cardio, sets/reps/added weight for Bodyweight). After each entry, calls alertManager.evaluate() to check for goal alerts. Ends when user enters 0. Prompts for overall session duration and notes before saving.
View recent sessionsLast 10 sessions sorted by date descending. Shows date, duration, number of exercises, and total session volume.
View session detailFull session detail: date, duration, notes, and each entry with exercise name, recorded values, and calculated volume.
Delete sessionConfirm before deletion.

Progress

Top exercisesList the top 10 most logged exercises by session entry count, showing total entries, total volume, and average volume per entry. Sorted by entry count descending.
Exercise progressionSelect an exercise and a date range. Display a table of every session entry for that exercise — date, recorded values, calculated volume — and an ASCII sparkline (▁▂▃▄▅▆▇█) of the key metric over time. Key metric is determined by exercise type: weightKg for Strength, durationMinutes ÷ distanceKm (pace) for Cardio, reps for Bodyweight.
Volume over timeTotal weekly training volume (sum of all entry volumes) plotted as an ASCII bar chart across the last 8 weeks. Highlights weeks where volume increased vs decreased.

Reports

Weekly summaryCurrent week: sessions completed vs goal, total volume, most trained muscle group, and longest session (highest total volume single workout). Triggers GoalProgressStrategy alert if behind on session target.
Monthly summarySelected month: total sessions, total volume, volume by exercise type (Strength / Cardio / Bodyweight), average session volume (total volume ÷ sessions completed), and top 3 most logged exercises.
Muscle group frequencyHow many times each muscle group was trained in a selected period. ASCII bar chart sorted by frequency. Highlights imbalances — e.g. chest trained 8× but back only 2×.

Settings

Options 4 and 5 are only displayed when Role.ADMINISTRATOR.equals(user.getRole()). The service layer additionally enforces this guard — UserService.addUser() and UserService.deleteUser() verify the current user's role via SessionContext and throw an UnauthorizedException if called by a non-administrator.

Update body weightPrompts for new body weight in kg. Updates the current user's bodyWeightKg field. Used by BodyweightExercise.calculateVolume().
Set weekly session goalPrompts for target sessions per week (default 3). Stored on the user. Referenced by GoalProgressStrategy during weekly report generation.
Change passwordPrompts for current password (verified), then new password twice. Hashed via Spring Security Crypto before persisting.
Add user (admin only)Prompts for name, username, and initial password. Role defaults to USER. Persists via UserRepository. Only displayed and executable by ADMINISTRATOR.
Delete user (admin only)Select a user from the list. Confirm before deletion. Cannot delete own account. Only displayed and executable by ADMINISTRATOR.
